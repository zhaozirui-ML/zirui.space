# AI Agent 协作规范 (AGENTS.md)

本文件定义项目与 AI 代理的长期协作规则。新规则与旧规则冲突时，以最新为准。

## 1. 沟通与代码风格
- **语言策略**：中文解释思路与注释（侧重说明“为什么”而非“做什么”）；英文用于变量、函数和组件命名。
- **沟通对象**：面向“非开发者”，必须解释推导过程，拒绝只给结论。
- **代码实现**：优先选择小范围、低风险、易回退的改动；实现前说明计划，实现后总结改动及作用；复杂实现补充解释性注释。

## 2. 工作流与变更控制
- **标准推进**：遵循“分析说明 -> 最小可行改动 -> 总结结果”的闭环；已有 dev server 时优先复用。
- **强阻断确认**：遇到以下情况，**必须**先说明原因与影响，等待确认后方可执行：
  - 新增依赖、外部 CDN 或涉及授权限制。
  - 较大或较复杂的实验性改动（需提醒新建分支）。
- **风险提示**：
  - 较大改动前必须输出：`⚠️ LARGE CHANGE ALERT`
  - 高风险修改前必须输出：`🔴 HIGH RISK MODIFICATION`

## 3. 工具链与依赖管理
- **包管理**：默认使用 `pnpm`。
- **Worktree 与依赖**：`git worktree` 新建工作目录后，先 `pnpm install` 再做其他事；如果 `typecheck` / `lint` / `dev` 报缺少命令，优先判断是不是当前 worktree 还没初始化。
- **自动检查**：代码修改后自动运行 `pnpm typecheck` 和 `pnpm lint`；若失败，需主动协助定位并修复。
- **Git 清理**：优先使用本地 `git-cleanup` skill；若手动执行，需确保删除前确认、保留受保护分支、汇报简洁。

## 4. 专项规则：字体接入
- **全局管理**：字体必须通过项目级全局方式（layout/变量/design token）正式接入，严禁在页面/组件中分散声明或依赖本机/临时字体。
- **临时与正式**：可先临时占位预览；一旦确认采用，需立即转为正式接入方案。引入新字体时需明确：来源、主要用途及影响范围。

## 5. 专项规则：颜色使用
- **禁止硬编码**：新增、修改或扩展颜色时，严禁直接使用 `#fff`、`#000`、`rgb()`、`rgba()` 等硬编码颜色值。
- **统一来源**：颜色必须优先使用 design system 或 portfolio theme 中已经存在的 color token / semantic token。
- **缺失处理**：如果现有 token 不适合当前语义，不要先写临时颜色；必须先说明缺少什么颜色语义，再决定是否补充到 design token。

## 6. 设计上下文文档分工
- 高层设计意图看 `docs/design-context.md`；系统层规则看 `DESIGN_SYSTEM.md`；页面层规则看 `GUIDELINES.md`。
- 如果三者出现直接冲突，不要扩大改动范围；先说明冲突点与影响，再等待确认后处理。

## 7. 专项规则：Figma 还原修复
- 当任务属于“从 Figma 设计稿还原后的页面细节修复”，并且用户在描述中提到 `对齐`、`还原`、`匹配`、`修复`、`不一致`、`偏了`、`裁切不对`、`高度/间距/位置不对`、`行为不一致` 这类问题时，优先调用 `$figma-fidelity-repair`。
- 当任务是第一次根据 Figma 实现页面时，继续使用 `$figma-implement-design`；当任务是已经实现后的细节校准时，切换到 `$figma-fidelity-repair`。
- 只要修复对象是 Figma 派生的视觉或交互细节，每次修复完成后都必须执行两项校验：
  - 对照 Figma 参考，检查最终 UI 的外观和行为是否一致
  - 使用 Playwright 检查浏览器里的实际渲染结果，并继续迭代直到结果稳定
- 如果两项校验中的任意一项仍不通过，不要把修复当作完成。
- 如果 Figma 参考不可用、图片资源无法稳定渲染，或者 Figma 与产品现状存在明确冲突，要先说明阻塞原因，再决定是否继续。
- 如果问题只影响单页，优先做页面层最小修复；只有确认多个页面共用同一错误时，才上移到共享组件。
