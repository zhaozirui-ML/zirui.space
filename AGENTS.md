# AGENTS.md

本文件用于记录这个项目对 AI 代理（如 Codex）的长期协作规则。

目标：
- 先建立一个清晰、可持续补充的模板。
- 后续你可以随时追加新规则，我会继续整理到对应章节。
- 如果新规则和旧规则冲突，以“最新补充的规则”为准，并尽量在相关章节中同步更新。

---

## 1. 项目协作方式

### 1.1 沟通原则
- 默认用中文解释思路、步骤和原因。
- 变量名、函数名、组件名使用英文。
- 面向“非开发者”说明，避免只给结论，不解释过程。
- 优先做小步修改，避免一次性大改。

### 1.2 变更原则
- 先做最小可行改动，再继续迭代。
- 如果变更较大、影响较广、或有潜在风险，先明确提醒，再继续实施。
- 需要新增依赖时，必须先说明“要装什么、为什么需要”，并等待确认。

### 1.3 规则维护方式
- 新增规则时，优先补充到最匹配的章节。
- 如果没有合适章节，可以新增小节，但保持结构清晰。
- 尽量把规则写成可执行、可检查的句子，避免模糊表述。

### 1.4 阶段记录规则
- “重要的阶段性改动”包括但不限于：
  - 新能力首次接入
  - 一段完整链路首次验证通过
  - 涉及多个文件或多个步骤的阶段收口
  - 后续大概率还会继续迭代的基础设施搭建
- 如果一项任务已经形成了明确的阶段成果，默认不要只停留在聊天总结里，应沉淀为可查阅的 MD 文件。
- 阶段 MD 默认归档到 `docs/phases/`，而不是直接放在项目根目录。
- 阶段 MD 的目标不是写成流水账，而是帮助未来快速回答这些问题：
  - 当时为什么这么做
  - 改动落在哪些文件
  - 哪些点已经验证过
  - 还有哪些是故意没做
  - 下一步最自然的延续动作是什么

### 1.5 默认工作流
- 默认按“小步修改”推进：
  - 先查看相关文件，再说明这次准备改什么
  - 先做最小可行改动，再继续下一步
  - 改动完成后，说明改了哪几处、每一处的作用是什么
- 关于新 worktree：
  - 如果某个 Thread 被放入新的 worktree，先判断当前任务是否真的需要启动项目、运行测试、执行 `lint` / `typecheck`，或其他依赖本地包的命令
  - 只有在任务确实需要运行环境时，才检查当前目录是否已经具备可运行依赖，而不是一进入新 worktree 就默认安装
  - 如果当前 worktree 缺少本地依赖，或关键命令不可用，默认先自动执行 `pnpm install`，再继续启动项目、调试或运行校验
  - 这里的 `pnpm install` 属于“安装当前项目已声明依赖到新 worktree”，默认不视为“新增依赖”，不需要再次单独确认
- 关于分支：
  - 小改动默认可以直接在当前分支进行
  - 较大改动、实验性改动、或风险较高的改动，默认先建分支
- 关于阶段 MD：
  - 小改动默认不强制写阶段 MD
  - 较大改动、跨多个模块的改动、或需要记录过程的任务，再写/更新阶段 MD
  - 只要是“重要的阶段性改动”，默认产出一个对应的阶段 MD，方便后续回顾、复习和继续迭代
  - 阶段 MD 默认放在 `docs/phases/` 目录下，文件名应清晰表达主题和阶段，例如：`docs/phases/supabase-storage-phase-1.md`
  - 阶段 MD 至少应包含：
    - 这次改动的目标
    - 这次做了什么
    - 这次没做什么
    - 改了哪些关键文件
    - 为什么这样改
    - 遇到了什么问题、怎么排查
    - 当前验证结果
    - 下一步建议
  - 如果本次任务本身就是“阶段收口”或“教学复盘”，默认在完成前同步写好对应阶段 MD
- 关于校验：
  - 代码改动后，默认运行 `pnpm typecheck` 和 `pnpm lint`
  - 如果改动影响页面展示，还需要检查对应页面是否正常
  - 如果本地已经有可用的 dev server，优先复用，不重复启动多个
- 关于完成后的回报：
  - 默认总结本次改动内容
  - 如果有分支或 commit，回报当前分支和最新 commit
  - 如果有阶段 MD，说明已更新哪个文件

## 2. 代码与实现偏好

### 2.1 改动风格
- 优先选择小范围、低风险、容易回退的修改方式。
- 在实现前，先说明本次准备改什么。
- 在实现后，说明改了哪几处、每一处的作用是什么。

### 2.2 注释与说明
- 代码注释使用中文。
- 注释重点说明“为什么这样做”，而不只是“这一行做了什么”。
- 对初学者不直观的实现，适当增加解释性注释。

### 2.3 包管理与检查
- 默认使用 `pnpm`。
- 完成代码修改后，自动运行：
  - `pnpm typecheck`
  - `pnpm lint`
- 如果检查失败，需要继续协助定位和修复问题。

### 2.4 字体接入规则
- 后续如果确定要使用新字体，默认把字体正式接入项目，而不是依赖本机已安装字体、临时 `font-family` 占位，或只停留在设计说明里。
- 字体应优先通过项目级统一方式接入和管理，例如全局字体入口、layout、字体变量或 design token，不要在页面或组件里分散重复声明字体来源。
- 非展示性文本默认统一使用 Satoshi，不要引入 `SF Mono`、系统 monospace 或其他临时功能字体。
- 这里的“非展示性文本”包括但不限于：导航、目录、按钮、标签、说明文字、表单文本、图注、辅助信息、数据说明。
- 展示性文本（如案例页主标题、章节标题、叙事型大标题）可以继续按页面规则使用展示字体，但功能性和信息性文字默认回到 Satoshi 体系。
- 接入字体时，需要同时说明：
  - 字体来源
  - 主要用途
  - 会影响哪些页面、模块或文字层级
- 如果字体需要新增依赖、外部 CDN、额外下载步骤，或可能涉及授权限制，必须先说明原因与影响，再继续实施。
- 如果只是快速预览或探索视觉方向，可以先临时占位；一旦确认采用，就应尽快改成正式接入方案，避免长期使用替代字体或仅在本机生效的写法。

## 3. 风险提示规范

- 较大改动前，使用 `⚠️ LARGE CHANGE ALERT` 明确提醒。
- 高风险修改前，使用 `🔴 HIGH RISK MODIFICATION` 明确提醒。
- 在进行较大或较复杂改动前，提醒用户先确认是否继续。
