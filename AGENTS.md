# AGENTS.md

本文件用于记录这个项目对 AI 代理（如 Codex）的长期协作规则。

目标：
- 先建立一个清晰、可持续补充的模板。
- 后续你可以随时追加新规则，我会继续整理到对应章节。
- 如果新规则和旧规则冲突，以“最新补充的规则”为准，并尽量在相关章节中同步更新。

---

## 1. 项目协作方式

### 1.1 沟通原则
- 默认用中文解释思路、步骤和原因。
- 变量名、函数名、组件名使用英文。
- 面向“非开发者”说明，避免只给结论，不解释过程。
- 优先做小步修改，避免一次性大改。

### 1.2 变更原则
- 先做最小可行改动，再继续迭代。
- 如果变更较大、影响较广、或有潜在风险，先明确提醒，再继续实施。
- 需要新增依赖时，必须先说明“要装什么、为什么需要”，并等待确认。

### 1.3 规则维护方式
- 新增规则时，优先补充到最匹配的章节。
- 如果没有合适章节，可以新增小节，但保持结构清晰。
- 尽量把规则写成可执行、可检查的句子，避免模糊表述。

### 1.4 阶段记录规则
- “重要的阶段性改动”包括但不限于：
  - 新能力首次接入
  - 一段完整链路首次验证通过
  - 涉及多个文件或多个步骤的阶段收口
  - 后续大概率还会继续迭代的基础设施搭建
- 如果一项任务已经形成了明确的阶段成果，默认不要只停留在聊天总结里，应沉淀为可查阅的 MD 文件。
- 阶段 MD 默认归档到 `docs/phases/`，而不是直接放在项目根目录。
- 阶段 MD 的目标不是写成流水账，而是帮助未来快速回答这些问题：
  - 当时为什么这么做
  - 改动落在哪些文件
  - 哪些点已经验证过
  - 还有哪些是故意没做
  - 下一步最自然的延续动作是什么

### 1.5 默认工作流
- 默认按“小步修改”推进：
  - 先查看相关文件，再说明这次准备改什么
  - 先做最小可行改动，再继续下一步
  - 改动完成后，说明改了哪几处、每一处的作用是什么
- 关于新 worktree：
  - 如果某个 Thread 被放入新的 worktree，先判断当前任务是否真的需要启动项目、运行测试、执行 `lint` / `typecheck`，或其他依赖本地包的命令
  - 只有在任务确实需要运行环境时，才检查当前目录是否已经具备可运行依赖，而不是一进入新 worktree 就默认安装
  - 如果当前 worktree 缺少本地依赖，或关键命令不可用，默认先自动执行 `pnpm install`，再继续启动项目、调试或运行校验
  - 这里的 `pnpm install` 属于“安装当前项目已声明依赖到新 worktree”，默认不视为“新增依赖”，不需要再次单独确认
- 关于分支：
  - 小改动默认可以直接在当前分支进行
  - 较大改动、实验性改动、或风险较高的改动，默认先建分支
- 关于阶段 MD：
  - 小改动默认不强制写阶段 MD
  - 较大改动、跨多个模块的改动、或需要记录过程的任务，再写/更新阶段 MD
  - 只要是“重要的阶段性改动”，默认产出一个对应的阶段 MD，方便后续回顾、复习和继续迭代
  - 阶段 MD 默认放在 `docs/phases/` 目录下，文件名应清晰表达主题和阶段，例如：`docs/phases/supabase-storage-phase-1.md`
  - 阶段 MD 至少应包含：
    - 这次改动的目标
    - 这次做了什么
    - 这次没做什么
    - 改了哪些关键文件
    - 为什么这样改
    - 遇到了什么问题、怎么排查
    - 当前验证结果
    - 下一步建议
  - 如果本次任务本身就是“阶段收口”或“教学复盘”，默认在完成前同步写好对应阶段 MD
- 关于校验：
  - 代码改动后，默认运行 `pnpm typecheck` 和 `pnpm lint`
  - 如果改动影响页面展示，还需要检查对应页面是否正常
  - 如果本地已经有可用的 dev server，优先复用，不重复启动多个
- 关于完成后的回报：
  - 默认总结本次改动内容
  - 如果有分支或 commit，回报当前分支和最新 commit
  - 如果有阶段 MD，说明已更新哪个文件

### 1.6 Figma 文案实现规则
- 展示文案的实现遵循 `GUIDELINES.md` 中的内容保真规则。

### 1.7 Figma 实现默认 Skills 组合
- 如果任务是“通过 Figma MCP 读取设计稿并实现为代码”，默认优先使用这组 Skills：
  - `figma`
  - `figma-implement-design`
  - `portfolio-design`
- 这组组合的分工默认理解为：
  - `figma`：负责正确读取 Figma 结构、截图、节点和资源
  - `figma-implement-design`：负责按设计稿高保真实现
  - `portfolio-design`：负责把实现结果拉回作品集项目自己的页面语言、叙事和版式规则
- 如果当前会话里 `portfolio-design` 不可用，默认退回：
  - `figma`
  - `figma-implement-design`
  - 同时继续遵循 `GUIDELINES.md` 和 `DESIGN_SYSTEM.md`
- 用户如果想直接调用，默认可用这类表达：
  - “请用 figma + figma-implement-design + portfolio-design 来实现这个设计稿”
  - “这次按 Figma MCP + portfolio-design 的方式来做”

### 1.8 Figma 截图卡片实现复盘规则
- 如果一张设计稿里出现“背景卡片 + 多张截图 / 视频 + 底部说明文案”的结构，默认不要直接套用通用媒体组件，先判断这块是不是一个“专用版式”。
- 对这类卡片，默认先区分清楚 3 层结构，再开始写代码：
  - 最外层 `figure`
  - 背景卡片本体
  - 卡片外部的 `figcaption`
- 不要先假设说明文案一定在卡片内部。必须先去 Figma 确认：
  - 文案是在卡片内部
  - 还是在卡片外部，并与卡片保持固定间距
- 不要先假设图片一定应该 `cover`。必须先判断：
  - 这是需要完整展示的参考拼图，应该优先 `contain`
  - 还是需要铺满裁切的局部视觉稿，应该使用 `cover`
- 对包含多张截图的卡片，不要先用“均分网格 + 自动居中”猜布局。优先读取 Figma 的真实信息：
  - 卡片高度
  - 截图尺寸
  - 距离卡片四边的 inset
  - 行距、列距
  - 左侧标签、顶部标签、底部说明文案的位置
- 如果 Figma 中的标签是“按行分组”的，例如“上线第一版 / 上线第二版”，默认不要给每张图单独挂标题，而是按 Figma 的真实结构实现成左侧行标签。
- 如果 Figma 中的图片区有背景纹理图、天空图、暖色底图等，默认把“背景图层”和“前景截图层”拆开实现，不要把前景截图直接当成唯一内容。
- 如果某个模块里的图片顺序带有语义，例如 `Before / After`、`原版 / 新版`，默认在实现后单独核对一次左右顺序，不要只检查标签文字。
- 对这类卡片，默认优先做“局部专用样式”，不要急着把它们强行抽象进通用样式里。只有在至少两个模块确认结构一致时，才考虑复用。
- 完成这类卡片后，默认做一次局部自检，不只看“有没有显示出来”，还要检查：
  - caption 是否在正确层级（卡片内 / 卡片外）
  - caption 与卡片之间的间距是否匹配 Figma
  - 卡片高度是否匹配 Figma
  - 图片的宽高是否匹配 Figma
  - 图片与卡片边缘的间距是否匹配 Figma
  - 多张图之间的行距和列距是否匹配 Figma
  - 标签、标题、说明文字的位置是否匹配 Figma
- 如果页面里已经有 Playwright，可默认用它做局部截图自检；自检时优先记录可量化数据，例如：
  - `frameHeight`
  - `imageWidth`
  - `imageHeight`
  - `imageInsetTop`
  - `imageInsetX`
  - `figureGap`

### 1.9 案例标题组件替换规则
- 如果项目里已经明确存在共享案例标题组件，例如 `CaseStudyHeadingOne / Two / Three`，默认先按“语义层级”直接替换，不要先额外发明新的视觉变体。
- 判断顺序默认是：
  - 先判断这是一级、二级还是三级标题
  - 再直接替换成对应层级组件
  - 替换后再去 Figma 复核样式
- 不要在没有 Figma 证据的情况下，先给同一层级标题补 `text`、`pill`、`accent` 之类的新变体。
- 只有在满足下面两个条件时，才考虑给同一层级组件增加新变体：
  - Figma 明确显示同一层级存在两种稳定样式
  - 至少两个真实页面都出现了同一种分叉，不是某一页的偶发现象
- 如果只是当前页面的某个标题样式看起来不对，默认先排查：
  - 是否用了错误层级组件
  - 是否正文或目录里的同名文本干扰了判断
  - 是否抓错了 Figma 节点
  而不是先扩组件 API。

## 2. 代码与实现偏好

### 2.1 改动风格
- 优先选择小范围、低风险、容易回退的修改方式。
- 在实现前，先说明本次准备改什么。
- 在实现后，说明改了哪几处、每一处的作用是什么。
- 如无必要，不新增实体层。能把样式直接挂在已有图片、文本、容器上的，默认不要额外包一层新的 DOM。
- 只有在新增那一层确实承担了独立职责时，才允许加新实体，例如：
  - 需要单独承载背景层
  - 需要单独控制裁切或遮罩
  - 需要单独负责定位、滚动、动画或交互命中区域
- 如果只是为了加描边、圆角、阴影、位移、透明度、颜色这类纯视觉样式，默认优先加在现有实体上，不要先叠新层。

### 2.2 注释与说明
- 代码注释使用中文。
- 注释重点说明“为什么这样做”，而不只是“这一行做了什么”。
- 对初学者不直观的实现，适当增加解释性注释。

### 2.3 包管理与检查
- 默认使用 `pnpm`。
- 完成代码修改后，自动运行：
  - `pnpm typecheck`
  - `pnpm lint`
- 如果检查失败，需要继续协助定位和修复问题。

### 2.4 字体接入规则
- 后续如果确定要使用新字体，默认把字体正式接入项目，而不是依赖本机已安装字体、临时 `font-family` 占位，或只停留在设计说明里。
- 字体应优先通过项目级统一方式接入和管理，例如全局字体入口、layout、字体变量或 design token，不要在页面或组件里分散重复声明字体来源。
- 字体选择遵循 `DESIGN_SYSTEM.md` 中的字体规则。
- 不要引入 `SF Mono`、系统 monospace 或其他临时功能字体。
- 展示性文本（如案例页主标题、章节标题、叙事型大标题）可以继续按页面规则使用展示字体，不要因为正文切换而一起改掉。
- 接入字体时，需要同时说明：
  - 字体来源
  - 主要用途
  - 会影响哪些页面、模块或文字层级
- 如果字体需要新增依赖、外部 CDN、额外下载步骤，或可能涉及授权限制，必须先说明原因与影响，再继续实施。
- 如果只是快速预览或探索视觉方向，可以先临时占位；一旦确认采用，就应尽快改成正式接入方案，避免长期使用替代字体或仅在本机生效的写法。

### 2.5 这轮截图卡片问题的共性
- 共性 1：把 Figma 里的“专用版式”误当成了“通用媒体卡片”，结果 caption、标签、网格、背景层级都容易放错位置。
- 共性 2：只验证了“内容有没有出来”，没有先验证“内容是不是在正确的层级里”，所以经常出现：
  - 文案跑到卡片内部
  - 文案本该在卡片内部却跑到外面
  - 标签被挂到每张图下面，而不是按行放在左边
- 共性 3：默认使用 `cover` 或默认铺满，会让参考拼图、总览图、收集图被裁掉。
- 共性 4：只看视觉大概像，不量具体尺寸，容易导致：
  - 图片和卡片边缘的间距不准
  - 行距 / 列距不准
  - 卡片高度不准
  - 图片顺序和语义不一致
- 共性 5：一次改太多块时，问题不容易定位。更稳的方式是一次只收一个模块，改完立刻局部自检。

## 3. 风险提示规范

- 较大改动前，使用 `⚠️ LARGE CHANGE ALERT` 明确提醒。
- 高风险修改前，使用 `🔴 HIGH RISK MODIFICATION` 明确提醒。
- 在进行较大或较复杂改动前，提醒用户先确认是否继续。
