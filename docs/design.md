# Design Notes

这份文档用于沉淀 Portfolio2026 在实际落地过程中反复验证过的页面实现策略。

它不替代以下文档：

- `docs/design-context.md`：高层设计意图、品牌气质、长期方向
- `DESIGN_SYSTEM.md`：token、组件、主题分层与职责边界
- `GUIDELINES.md`：页面层模式、内容规则、Figma 落地原则

这份文档更偏“实现决策备忘录”：

- 哪些问题应该优先在页面层解决
- 哪些文本与排版策略已经形成稳定规则
- `harden` / `polish` 这类收尾阶段，应该优先检查什么

---

## 1. 页面层优先级

### 1.1 什么问题优先在页面层修

如果问题属于下面这些类型，优先在页面层做最小修复，不要第一反应就改 base design system：

- 首页卡片排版节奏
- 页面专用标题换行
- 页面专用摘要宽度
- 页面专用 hover / focus 细节
- 某个 section 的响应式特例

原因：

- 这些问题通常是作品集页面的版式语义，不是基础组件能力缺陷
- 直接改基础组件，容易把一个页面特例误提升成全局规则
- 页面层修复更容易控制影响面，也更容易回退

### 1.2 什么情况下再往上提

只有在确认多个页面共享同一种错误时，才考虑上移到：

- portfolio theme layer
- shared page styles
- base design system

不要因为一个页面里重复出现两三次，就过早抽成全局规则。

## 2. 文本与排版策略

### 2.1 展示型短文案默认不带句号

以下类型默认优先不带句号：

- 首页卡片辅助文本
- Blog / Work 列表页卡片摘要
- Skills caption
- 简历经历 bullet
- 短列表项

原因：

- 这些内容是 UI 辅助信息，不是正文段落
- 列表结构、卡片结构本身已经承担“分句”作用
- 去掉句号后更克制，也更符合当前作品集的 editorial 气质

### 2.2 正文型文案保留自然标点

以下内容不要为了“统一格式”强行去标点：

- 文章正文
- 案例正文
- detail summary
- 长段落说明
- 引语、带语气的句子

原因：

- 这些内容属于阅读型正文
- 标点本身是语义的一部分，不是纯视觉样式

### 2.3 不要做“中文不带、英文带”的机械规则

对于 bullet、caption、summary 这类展示型短文案：

- 优先按“文案类型”定规则
- 不要按“语言”硬拆规则

更好的统一方式是：

- 同一种 UI 语义，中文和英文都遵循同一规则

例如：

- 首页卡片摘要：中英文都不带句号
- 简历 bullet：中英文都不带句号

## 3. 中文标题换行策略

### 3.1 不要默认给中文标题使用 `text-wrap: balance`

`text-wrap: balance`` 在英文标题上通常效果不错，但在中文大标题上容易出现：

- 明明右边还有空间，却提前换行
- 标题没有吃满容器宽度
- 视觉上显得“断得很早”

因此：

- 英文标题可以继续用 `balance`
- 中文标题优先用更自然的换行策略，例如 `pretty`

### 3.2 这是定向规则，不是全站一刀切

不要一次把所有标题都改成同一种换行方式。

正确做法是：

- 先确认问题是否真实出现
- 再按具体标题类型定向修复

已经验证适合这套策略的区域包括：

- 首页作品卡片标题
- Blog 列表页 featured 标题
- Blog 列表页 browse 卡片标题
- Work 列表页项目标题

## 4. 摘要宽度与文本容器策略

### 4.1 桌面端可以保留阅读宽度限制

对于卡片摘要、辅助文本：

- 桌面端可以保留 `max-width`
- 这样能避免一行过长，阅读更稳定

### 4.2 平板和移动端不要盲目沿用桌面 `max-width`

如果页面在平板 / 手机上已经切成单列：

- 摘要文本通常应该跟随内容容器铺满
- 不应该继续被桌面端的固定 `max-width` 提前限制

否则会出现：

- 文本看起来像“没填满容器”
- 明明还有空间，却提前换行
- 信息层级显得发虚、不稳定

因此：

- 桌面端的 `max-width` 可以保留
- 单列断点下通常应放开 `max-width`

## 5. Hover / Focus / Touch 输入方式策略

### 5.1 hover 只属于真正支持 hover 的设备

所有依赖 hover 的视觉反馈，都应优先限制在：

```css
@media (hover: hover) { ... }
```

适用场景：

- 卡片浮起
- 边框强化
- 图标滑入
- 链接 hover 变色
- 头像放大

原因：

- 触屏设备不应吃到桌面 hover 逻辑
- 否则会出现 hover 残留、交互发黏、反馈不自然

### 5.2 focus 不能依赖 hover 一起实现

`focus-visible` 应单独定义，不要只靠 `:hover, :focus-visible` 混写。

原因：

- 键盘导航需要稳定、明确、可预期的反馈
- 触屏环境不应因为 hover 复用而被牵连

推荐做法：

- hover：仅在 `hover: hover` 设备上生效
- focus-visible：始终保留，单独定义轮廓、颜色、位移或图标显现

### 5.3 触屏设备上要主动撤掉假 hover

对于已经写了桌面 hover 的元素，如果触屏环境会误触发：

- 可以在 `@media (hover: none)` 下明确撤销 hover 结果

适用场景：

- Work side projects
- 列表型卡片
- 图标显隐类细节

## 6. 触控尺寸与移动端稳健性

### 6.1 移动端最小点击高度优先保证 44px 左右

对于以下高频交互：

- CTA 按钮
- tabs
- 导航按钮
- 语言切换
- 图标按钮

在移动端优先保证更稳的点击尺寸，不要沿用桌面的小高度。

### 6.2 不要只看视觉尺寸，要看可点击尺寸

常见误区：

- 看起来按钮不小，但实际 `min-height` 太紧
- 标签文字可以点，但外围 padding 不够

实际检查时，优先看：

- `min-height`
- `padding`
- 相邻点击目标是否太密

## 7. 当前已经验证有效的修复模式

### 7.1 文本问题

如果再遇到下面两类问题：

1. 文本没填满容器
2. 中文标题过早换行

优先按这个顺序排查：

1. 是否有桌面端遗留的 `max-width`
2. 是否使用了 `text-wrap: balance`
3. 是否是 grid / flex 子项没有 `min-width: 0`
4. 是否是 line clamp 提前截断

### 7.2 修复策略

优先使用：

- 断点定向修复
- `:lang(zh)` 定向修复
- 页面层修复
- hover / focus 分离

避免：

- 一上来改全局 token
- 一上来改基础组件
- 一上来全站替换某个 CSS 策略

## 8. `harden` 与 `polish` 的执行顺序

### 8.1 先 `harden`

先修这些：

- 触控热区
- 焦点态
- 长文本边界
- 触屏 hover 误用
- 响应式断点下的布局稳健性

### 8.2 再 `polish`

再修这些：

- 阴影
- 对齐
- 间距节奏
- 链接与图标的显隐细节
- 排版细微不一致

原因：

- 如果先 polish，再 harden，后面的稳健性修复很容易打乱刚刚收好的视觉细节

---

## 9. 后续继续检查时的优先顺序

后面如果继续做第 4 点和第 5 点，优先检查：

1. 站点级导航与底部悬浮导航
2. 首页 Recent Works / Blogs / Skills
3. Work 列表页 tabs / project cards / side projects
4. About 页外链、SNS、meta 区

暂时不要第一时间扩到所有案例详情页，除非确认问题已经跨页面重复出现。
