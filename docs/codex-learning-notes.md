这份文档不是阶段记录，也不是设计规范。

它的目标是把这段时间协作中出现过、当时你不太确定的知识点整理成一份可复习的学习笔记，方便后面继续做项目时快速回看。

## 1. Skills、Plugin、MCP 分别是什么

这三个词很容易混在一起，但它们不是同一层东西。

### 1.1 MCP 是什么

MCP 可以理解成一种“能力接口”。

它负责让我拿到某个外部工具或系统的能力，比如：

- 读取 Figma 设计稿
- 获取 GitHub PR 信息
- 控制浏览器

在这个项目里，常见的 MCP 能力有：

- Figma MCP
- Playwright MCP

简单理解：

- MCP 更像“工具通道”
- 它回答的是：**我能连接到什么能力**

### 1.2 Skill 是什么

Skill 不是一个外部工具，而是一套“工作方法”。

它更像一份针对某类任务的操作说明，比如：

- 先做什么
- 后做什么
- 哪些步骤不能跳
- 应该优先用哪些工具

例如：

- `figma`
- `figma-implement-design`
- `interface-craft`

简单理解：

- Skill 更像“做事流程和方法论”
- 它回答的是：**我应该怎么用这些能力去完成任务**

### 1.3 Plugin 是什么

Plugin 可以理解成一个“能力包”。

它通常会带来：

- 一组 MCP 工具
- 一组相关 Skill

比如：

- `Figma` plugin
- `GitHub` plugin

简单理解：

- Plugin 更像“功能套装”
- 它回答的是：**这个插件整体能帮我做哪一类事**

### 1.4 一句话区分

- MCP：能力接口
- Skill：做事方法
- Plugin：把能力和方法打包到一起的套装

## 2. 什么时候该用哪个 Skill

### 2.1 和 Figma 设计稿实现最相关的 Skill

当前这个项目里，最常用的组合是：

- `figma`
- `figma-implement-design`
- `Portfolio Design`

这组组合在 `AGENTS.md` 里已经沉淀成规则。

它们的分工是：

- `figma`
  - 负责正确读取 Figma 结构、截图、节点和资源
- `figma-implement-design`
  - 负责高保真实现设计稿
- `Portfolio Design`
  - 负责把实现结果拉回当前作品集项目自己的气质、叙事和版式规则

简单理解：

- `figma` 负责“看清设计稿”
- `figma-implement-design` 负责“把设计稿实现出来”
- `Portfolio Design` 负责“让实现结果像你的作品集，而不是像一个普通页面”

### 2.2 之前讨论过的几个交互 Skill 差异

这几个 Skill 之前主要用在目录交互优化里。

#### `animate`

适合：

- 你已经知道要加动效
- 重点是把动画做顺

更关注：

- easing
- duration
- hover / active / enter / exit 的流畅度

简单理解：

- 它更像“专门做动画细节的 Skill”

#### `interface-craft`

适合：

- 你不只是要动效
- 还想让交互更像一个完整的设计语言

更关注：

- 节奏感
- 分层感
- 哪些该动，哪些不该动
- 是否有设计感

简单理解：

- 它更像“交互质感导演”

#### `make-interfaces-feel-better`

适合：

- 页面已经能用
- 但你觉得“感觉不太对”

更关注：

- 边框轻重
- 阴影是否太重
- hover 力度是否过度
- 文字、图标、圆点、间距的细微手感

简单理解：

- 它更像“界面手感修整器”

#### `polish`

适合：

- 页面大方向已经定了
- 现在做最后一轮统一收尾

更关注：

- 对齐
- 间距
- 一致性
- 小瑕疵

简单理解：

- 它更像“交付前的精修”

### 2.3 为什么这些 Skill 可以组合使用

因为它们解决的问题层级不同。

例如：

- `interface-craft`
  - 决定交互方向
- `make-interfaces-feel-better`
  - 调手感
- `polish`
  - 做最后统一检查

所以关键不是“能不能混用”，而是：

**不要让多个 Skill 同时负责同一件事。**

## 3. 什么叫“站点层”“页面层”“组件层”

这是前面字体规则、布局规则里反复出现的概念。

### 3.1 站点层

站点层可以理解成：

**整个网站最外层的公共外壳。**

它通常负责：

- 整个站点默认字体
- 全局背景色
- Header / Footer
- 页面主区域的外层结构

在这个项目里，典型文件是：

- `src/site/components/SiteLayout.jsx`
- `src/site/styles/site-shell.module.css`

### 3.2 页面层

页面层是：

**某一个页面自己的布局和内容组织。**

例如：

- `HomePage.jsx`
- `WorkIndexPage.jsx`
- `WorkDetailPage.jsx`

它更关注：

- 这一页的 section 怎么排
- 哪些内容在这一页出现
- 页面自己的节奏

### 3.3 组件层

组件层是：

**页面里一块块可复用的内容模块。**

例如：

- `CaseStudyToc`
- `DrawingLedgerCaseStudy` 里的某些媒体块

它更关注：

- 这一块内容怎么实现
- 这一块能不能在别处复用

### 3.4 一个很好记的判断方式

- 改默认字体、全局 header、footer：站点层
- 改某一页 section 间距：页面层
- 改目录、卡片、图片区：组件层

## 4. 什么是类（Class）

这里说的 Class，通常指 CSS class，也就是样式类名。

例如：

```css
.detailPage {
  font-family: var(--portfolio-font-body);
}
```

这里的 `.detailPage` 就是一个 class。

在 React / Next 组件里，通常会这样用：

```jsx
<div className={styles.detailPage}>...</div>
```

意思是：

- 给这个元素挂上一个样式类
- 这个元素就会吃到 `.detailPage` 对应的样式

### 为什么 class 很重要

因为它是“规则落地的位置”。

比如之前做“详情页正文 = Inter”时，如果只是口头说规则，没有 class，这个规则就很难被稳定复用。

后来我们做的事情就是：

- 给详情页抽出一个更明确的页面级类名 `detailPage`

这样后面第 2、3 个项目详情页接进来时，只要也挂这个类，就能复用同一条字体规则。

## 5. 为什么新 worktree 里经常要重新装 `node_modules`

这是你之前问过的一个非常实用的问题。

### 5.1 根本原因

worktree 虽然来自同一个 Git 仓库，但它本质上是：

**另一个独立目录。**

例如：

- 原目录
- 新 worktree 目录

对 Node / Next 来说，它们是两个不同的项目目录。

而 `node_modules`：

- 不会被 Git 提交
- 也不会自动跟着 worktree 一起复制出来

所以新 worktree 往往只有源码，没有安装结果。

### 5.2 为什么又不会每次都特别慢

因为 `pnpm` 有共享 store。

所以很多时候，新 worktree 执行 `pnpm install` 并不是重新全量下载，而是：

- 检查依赖
- 从本机缓存里快速链接过来

### 5.3 当前项目已经沉淀出的规则

现在项目规则已经写进 `AGENTS.md`：

- 新 worktree 不是一进来就默认安装依赖
- 先判断任务是否真的需要运行项目 / lint / typecheck
- 只有需要运行环境时，才检查并补齐依赖

## 6. `AGENTS.md`、`GUIDELINES.md`、`DESIGN_SYSTEM.md` 分别放什么

这三个文档也很容易混。

### 6.1 `AGENTS.md`

放的是：

- 协作规则
- 执行规则
- AI 代理做事时该遵守什么

例如：

- 改完代码后跑什么检查
- 新 worktree 时怎么处理依赖
- Figma 实现时默认用哪些 Skill

简单理解：

- `AGENTS.md` 是“怎么协作”

### 6.2 `GUIDELINES.md`

放的是：

- 页面层规范
- 作品集页面应该是什么气质
- section、媒体框、图注、内容保真等页面规则

例如这次新放进去的：

- Figma 展示文案按原文逐字实现，不擅自补标点

简单理解：

- `GUIDELINES.md` 是“页面应该长什么样、怎么表达”

### 6.3 `DESIGN_SYSTEM.md`

放的是：

- 设计系统本体
- token
- 字体规则
- 色彩
- 间距
- 组件归属

例如：

- 模块首页正文 = Satoshi
- 项目详情页正文 = Inter

简单理解：

- `DESIGN_SYSTEM.md` 是“系统级视觉规则”

### 6.4 一个简单记忆法

- `AGENTS.md`：协作规则
- `GUIDELINES.md`：页面规范
- `DESIGN_SYSTEM.md`：视觉系统规则

## 7. 为什么 Supabase 接入要先测一页，再逐步推广

你之前问过：

- 要不要等 3 个项目都做完再一起接入 Supabase
- 还是先接一个项目试点

最后选择的是：

**先拿当前项目做试点，再推广到后面的项目。**

### 原因

如果一口气同时接 3 个项目：

- 出问题时很难排查
- 不知道是路径错、命名错，还是某一页资源漏传

如果先接 1 个项目：

- 先验证 bucket、路径、代码切换、图片显示链路
- 后面两个项目就可以复用这套方法

### 这个项目里我们已经做的准备

在当前案例页里，资源层已经拆成了独立映射文件：

- `src/site/data/work-details/drawing-ledger-2-0.js`

这里把：

- 本地路径
- Supabase 路径

都集中管理了，所以页面组件本身不用重写。

## 8. 为什么这次 Supabase 图片上传成功了，但页面里不显示

这个问题很关键，因为它说明“上传成功”和“页面能显示”不是同一件事。

### 8.1 这次真实的根因

不是：

- 图片没上传
- bucket 没公开
- 文件损坏

而是：

- Supabase 图片本身是好的
- 但 Next 的 `/_next/image` 优化接口把这类远程 URL 拦掉了

我们后来实际测到的报错是：

- `"url" parameter is not allowed`

### 8.2 这说明了什么

说明一条资源链路至少有两层：

1. 图片本身能不能被访问
2. 页面框架愿不愿意按自己的方式去处理这张图

这次是第 1 层没问题，第 2 层出了问题。

### 8.3 后来怎么解决

不是全局关掉 Next 图片，而是：

- 只让这页来自 Supabase 的图片绕开 `/_next/image`
- 直接用 Supabase 的公开地址渲染

这就是一个典型例子：

**定位问题时，要区分“资源问题”和“框架处理问题”。**

## 9. 目录为什么会“点了像没选中”

这个问题也很典型，适合复习。

### 表面现象

- 点击目录项
- 页面跳过去了
- 但当前项没有稳定高亮

### 根因

之前目录组件的选中态完全依赖 `IntersectionObserver`。

也就是说：

- 点击之后，不会立即把当前项设为 active
- 它要等滚动观察器自己判断“哪个标题更靠近顶部”

如果几个标题彼此很近，就可能出现：

- 你点了 A
- 但观察器觉得 B 离顶部更近
- 所以看起来像“没点中”

### 修复思路

后来做的最小修复是：

- 点击目录项时，先立即设置 active
- 再继续保留滚动观察逻辑

这也是一个很好记的前端经验：

**用户点击触发的状态，最好先给即时反馈；不要完全等被动观察逻辑来决定。**

## 10. 为什么目录层级后来要从 `level: 0/1` 改成语义字段

这件事的核心不是“技术更高级”，而是：

**更不容易写错。**

之前目录项是这样写的：

```js
{ id: "case-goal", label: "设计目标", level: 0 }
```

问题是：

- `0` 是一级
- `1` 是二级

每次看都要脑内翻译一次，特别容易写错。

后来改成了：

```js
{ id: "case-goal", label: "设计目标", hierarchy: "primary" }
```

这样好处是：

- 一眼能看懂
- 比 `0 / 1` 更有语义
- 后面复用到别的项目时更稳

这类改动通常叫：

**把数据表达方式从“脆弱数字”改成“更有语义的字段”。**
