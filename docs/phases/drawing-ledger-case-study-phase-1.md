# 图纸台账案例页 Phase 1

## 这次改动的目标

- 把 `/work/drawing-ledger-2-0` 从占位骨架升级成真实可浏览的案例长页
- 尽量按 Figma 长页结构落地封面、背景、问题、设计实践、细节展开、移动端设计、结果与反思
- 提前把图片资源接入方式设计成“后续可平滑切到 Supabase Storage”的结构，而不是把资源路径散落在 JSX 里

## 这次做了什么

- 新增了图纸台账案例页组件，替换原本详情页里的占位说明
- 新增了专用样式模块，把页面层排版、媒体框、图注、任务分析表、横向滚动卡片等都独立出来
- 新增了案例页目录功能，并完成了一轮样式重构
  - 先做了最小可用版目录，验证锚点跳转和当前章节高亮
  - 随后根据新反馈，把目录改成更接近“文档目录”的左侧样式
  - 当前目录已支持一级标题 + 二级小节锚点，并加入更克制的 hover 动效
- 新增了案例页图片资源映射文件
  - 当前默认先读取本地 `public/site/work/drawing-ledger-2-0/` 下的静态图
  - 同时为每张图预留了未来的 `storagePath`
  - 后面切到 Supabase 时，主要改资源映射层即可
- 下载并接入了大部分 Figma 图片资源，确保页面当前能正常展示主视觉和多数关键截图
- 对于 Figma MCP 这次没有直接吐出资源链接的几张图，先保留了正式布局位置，并放入“待接入资源”的说明卡，避免后续重排布局

- 还没有把资源映射层切换成真正从 Supabase 读取
- 还没有为“流程演示图”“响应式演示图”“移动端交互演示图”补上最终正式图片

- 还没有把目录继续升级成更强的“阅读进度型”侧栏
- 还没有针对目录做更细的视觉验收，例如桌面定位、不同视口下的密度和 hover 手感

## 为什么这样改

- 这次页面是长页案例，不适合继续塞在通用 `WorkDetailPage` 占位结构里，所以单独抽成案例组件更清晰
- 样式单独拆成模块，是为了把“页面层排版”和“其他站点页样式”隔开，避免互相污染
- 资源映射文件单独存在，是为了后续切 Supabase 时只改资源入口，不需要重新翻整页 JSX
- 本地静态图先接进来，是因为当前 Figma MCP 的资源链接是临时的，不适合直接作为正式站点长期依赖
- 目录之所以从顶部条改成左侧文档型样式，是因为这个案例页更像一篇长文档，左侧弱化式导航更符合阅读心智，也更接近参考案例的感觉

## 遇到了什么问题、怎么排查

### 1. Figma 资源链接不能直接作为长期正式资源

- 问题：Figma MCP 返回的是临时资产 URL，后续会失效
- 处理：先下载成本地静态图，并在资源映射层同时预留 Supabase 的 `storagePath`

### 2. 某几张图在 Figma MCP 中没有直接返回正式资源

- 问题：流程演示图、响应式演示图、移动端交互演示图在这次 MCP 返回里没有直接拿到最终图片资产
- 处理：先保留完整布局和说明文案，用待接入占位卡顶住结构，避免以后重新排版

### 3. 本地自动截图工具环境异常

- 问题：Playwright MCP 需要在根目录创建 `/.playwright-mcp`，当前环境是只读，无法生成目录
- 处理：改用本地开发服务返回的 HTML 做结构确认，并记录这次没能完成自动截图验证

### 4. `pnpm typecheck` 提示组件参数类型不完整

- 问题：`checkJs` 下，组件参数里没给默认值的字段会被推断成必填
- 处理：给 `TitleHeader` 和 `MediaFigure` 的可选参数补了默认值，随后重新通过类型检查

### 5. 长页信息太多，阅读时容易迷路

- 问题：当前案例页很长，如果没有章节导航，阅读和回看成本都偏高
- 处理：先增加最小目录验证锚点逻辑，再进一步补齐二级锚点，最后把样式重构成左侧文档型目录
