# Style Systemization Checklist

这份文档记录本项目当前已经跑通的“样式系统化”判断方式。

它的目标不是把所有 CSS 数值都抽掉，而是帮助你快速判断：

- 哪些值应该进入 token 或语义变量
- 哪些值应该保留为页面特例
- 哪些值是构图专用，不适合继续抽象

## 1. 优先系统化的值

这些值通常值得统一：

- 全站壳层宽度、左右留白、底部安全距离
- 页面 section 之间的纵向节奏
- 常见 card / panel / placeholder 的 padding
- 常见卡片圆角
- 导航、按钮、标签、图注、列表的重复间距
- 响应式断点里反复出现的收缩值

判断标准：

- 同类样式在多个文件里重复出现
- 改一次会影响多个页面
- 值本身表达的是“规则”，不是“构图”

## 2. 适合保留的页面特例

这些值不建议强行统一：

- 案例页里精确的 absolute 定位
- 媒体卡片里为了对齐 Figma 的 inset 和比例
- 只服务单个模块的局部偏移
- 只在一个断点里出现、且不复用的微调值
- 明显属于视觉构图的一次性数值

判断标准：

- 去掉后会破坏页面构图
- 值本身没有跨页面复用价值
- 它描述的是“这个页面怎么摆”，不是“全站怎么摆”

## 3. 当前项目里已经统一得比较好的层

这些地方已经比较适合继续沿用当前写法：

- `site-shell.module.css`
- `home-page.module.css`
- `about-page.module.css`
- `blog-page.module.css`
- `blog-detail-page.module.css`
- `case-study-heading.module.css`
- `case-study-toc.module.css`
- `data-visualization-screen-detail.module.css`
- `drawing-ledger-case-study.module.css`
- `axzo-design-system-case-study.module.css`

这些文件现在的共同点是：

- 页面节奏更清楚
- 高复用尺寸更集中
- 移动端断点更容易维护

## 4. 一个简单的判断流程

当你看到一个写死数值时，先按这个顺序判断：

1. 这个值是不是在两个以上页面里重复出现？
2. 这个值是不是表达了通用规则，而不是某个构图细节？
3. 这个值改动后，会不会影响多个地方？
4. 如果把它抽成变量，会不会让代码更好读？

如果前 3 个答案里有 2 个以上是“是”，通常就值得系统化。

如果答案大多是否，而且这个值明显跟某个页面构图强绑定，就保留为页面特例。

## 5. 以后优先统一的方向

后续如果继续整理，优先顺序建议是：

- 全站壳层和导航
- 首页和 about / blog 这种高复用页面
- 案例页里重复的 section rhythm
- 最后才是单页的构图细节

## 6. 不建议继续硬抽的东西

这些通常可以停：

- 很细的绝对坐标
- 明显依赖内容宽高的 inset
- 特定图块的视觉摆位
- 单个页面的实验性动效参数

原因很简单：

- 这些值抽成变量后，语义会变差
- 后面的人还得反向理解“这个变量到底服务什么”
- 维护成本会比收益更高

## 7. 当前结论

这次推进已经把项目从“散落数值”推进到“局部语义变量 + token”的状态。

现在更重要的不是继续消灭每一个数字，而是保持边界清楚：

- 规则性的东西继续统一
- 构图性的东西保留特例
- 不要为了系统化而系统化
